# Лабораторная работа №1–4

## Введение
Данный проект представляет собой расширенную реализацию системы для работы с задачами. В отличие от предыдущих версий, здесь добавлены:
- **Очередь задач** с поддержкой итераторов, генераторов и ленивых фильтров (ЛР3);
- **Асинхронный исполнитель** задач с контрактными обработчиками и контекстными менеджерами (ЛР4).

Проект демонстрирует углублённое применение принципов SOLID, работу с исключениями, продвинутое логирование, контрактное программирование, дескрипторы, итераторы, генераторы, а также асинхронное программирование на базе `async`/`await`.

## Описание компонентов

### models (Модели данных)

- `descriptors.py`: Содержит классы-дескрипторы для управления атрибутами модели `Task`.
  - `ValidatedString`: Data descriptor для строк с проверкой длины.
  - `PositiveInteger`: Data descriptor для целых чисел в заданном диапазоне.
  - `PriorityDescriptor`: Специализированный дескриптор для приоритета, возвращающий словарь со значением и названием уровня (Low, Medium, High, Critical).
  - `StatusDescriptor`: Non-data descriptor для статуса с проверкой допустимых переходов (pending → running → completed/canceled).
  - `TimestampDescriptor`: Data descriptor для временных меток с автоматической установкой.
  - `DataDescriptor`: Non-data descriptor для доступа к полезной нагрузке (`payload`).

- `exceptions.py`: Определяет иерархию исключений для валидации.
  - `TaskValidationExcept`: Базовое исключение.
  - `InvalidPriorityError`, `InvalidStatusError`, `InvalidDescriptionError`, `InvalidIDError`: Специализированные исключения.

- `task.py`: Содержит класс `Task`, который использует дескрипторы для атрибутов.
  - **Атрибуты**: `id`, `description`, `priority`, `created_at`, `status`, `payload`.
  - **Вычисляемые свойства**: `is_ready`, `age`, `age_formatted`, `priority_name`, `is_completed`.
  - **Методы**: `upd_status()` для безопасного изменения статуса, `to_dict()` и `from_dict()` для сериализации.

- `queue.py`: Содержит класс `TaskQueue` — очередь задач с поддержкой итерации и ленивой фильтрации (ЛР3).
  - **Базовые операции**: `push()`, `pop()`, `peek()`, `size()`, `is_empty()`, `clear()`, `extend()`, `to_list()`, `copy()`.
  - **Итерация**: Поддержка протокола итерации (`__iter__`, `__next__`) с возможностью повторного обхода очереди.
  - **Ленивые фильтры** (генераторы):
    - `filter_by_status()` — фильтрация задач по статусу.
    - `filter_by_priority()` — фильтрация по диапазону приоритета.
    - `filter_by()` — фильтрация с пользовательским предикатом.
  - **Поддержка стандартных конструкций**: `len()`, `bool()`, `for`, `list()`.

### src (Исходный код приложения)

- `contracts.py`: Содержит базовые контракты системы (датакласс `Task` и протокол `TaskSource`), а также функцию `check_task_source` для проверки соответствия протоколу.
- `logger_config.py`: Настраивает логирование в консоль и файл с динамическим именем.
- `sources.py`: Реализации источников задач (`FileTaskSource`, `GeneratorTaskSource`, `APITaskSource`).
- `processor.py`: Класс `TaskProcessor` для управления источниками и обработки задач.
- `handler.py`: Содержит контракт `TaskHandler` (Protocol) и реализации обработчиков (`PrintHandler`, `PriorityHandler`, `FailingHandler`) для ЛР4.
- `async_executor.py`: Класс `AsyncTaskExecutor` — асинхронный исполнитель задач с очередью, пулом воркеров, контекстным менеджером и централизованной обработкой ошибок.
- `main.py`: Точка входа, демонстрирующая работу всех четырёх лабораторных работ (источники, модель с дескрипторами, очередь задач, асинхронный исполнитель).
- `old_main.py`: Предыдущая версия `main.py` (ЛР1–3), сохранённая для истории.

### tests (Тесты)

- `test_contracts.py`: Проверка протокола `TaskSource` и функции проверки.
- `test_sources.py`: Тестирование каждого источника задач.
- `test_processor.py`: Юнит-тесты для класса `TaskProcessor`.
- `test_queue.py`: Тесты для очереди задач `TaskQueue` (итерация, фильтры, эффективность).
- `test_async_executor.py`: Тесты для асинхронного исполнителя (воркеры, обработчики, ошибки, контекстный менеджер).
- `test_integration.py`: Интеграционные тесты, проверяющие совместную работу всех компонентов.

## Демонстрация возможностей

### 1. Работа с источниками задач (ЛР1)
- **Контрактное программирование**: Все источники реализуют протокол `TaskSource` с методом `get_tasks()`.
- **Типы источников**: файловый (`FileTaskSource`), генераторный (`GeneratorTaskSource`), API-заглушка (`APITaskSource`).
- **Процессор задач**: `TaskProcessor` управляет источниками и собирает задачи через единый контракт.

### 2. Модель задачи с дескрипторами (ЛР2)
- **Валидация**: При создании задачи проверяется длина `id` и `description`, значение `priority` (1-4) и допустимость начального `status`.
- **Управление статусом**: Метод `upd_status()` контролирует переходы между статусами, запрещая некорректные изменения.
- **Автоматические поля**: `created_at` устанавливается автоматически при создании задачи.
- **Вычисляемые свойства**: `age_formatted` возвращает удобочитаемый возраст задачи, `priority_name` — название уровня приоритета.
- **Data vs Non-data descriptors**: Демонстрация различий между дескрипторами с хранением данных и без.

### 3. Очередь задач: итераторы и генераторы (ЛР3)
- **Итерация**: Поддержка протокола итерации с возможностью повторного обхода очереди.
- **Ленивые фильтры**: Фильтрация по статусу, приоритету или пользовательскому предикату с использованием генераторов (без создания копий в памяти).
- **Эффективность**: Работа с большими объёмами данных (10000+ задач) без избыточного потребления памяти.
- **Стандартные конструкции**: Поддержка `len()`, `bool()`, `for`, `list()`.

### 4. Асинхронный исполнитель задач (ЛР4)
- **Асинхронная очередь**: Использование `asyncio.Queue` для потокобезопасной передачи задач.
- **Контракт обработчика**: Протокол `TaskHandler` с `@runtime_checkable` для проверки соответствия во время выполнения.
- **Пул воркеров**: Параллельная обработка задач с настраиваемым количеством воркеров.
- **Контекстный менеджер**: Поддержка `async with` для автоматического запуска и остановки исполнителя.
- **Обработка ошибок**: Централизованное логирование ошибок с сохранением статистики успешных и упавших задач.
- **Расширяемость**: Новые обработчики добавляются без изменения существующего кода — достаточно реализовать протокол `TaskHandler`.

### 5. Интеграция
- Совместная работа источников задач (ЛР1), модели `Task` (ЛР2), очереди `TaskQueue` (ЛР3) и асинхронного исполнителя (ЛР4).
- Получение задач из источников, преобразование в доменные задачи, помещение в очередь, асинхронная обработка с логированием и сбором статистики.

### 6. Логирование
- Все операции логируются в консоль и в файл с меткой времени.
- Уровень логирования настраивается в `main.py` (см. раздел «Запуск проекта»).

## Требования к коду и инструментарий

При разработке данного проекта соблюдались следующие правила и использовались инструменты:

- **Стиль кода**: Код написан в соответствии с PEP8.
  - `snake_case` для имён переменных, функций и модулей.
  - `PascalCase` для имён классов.
  - Отступы: 1 шаг = 4 пробела.
- **Документирование**: Код содержит `docstring` для классов и функций, а также аннотации типов (`type hints`).
- **Логирование**: Используется модуль `logging` с выводом в консоль и файл.
- **Дескрипторы**: Для валидации и управления атрибутами применены data и non-data дескрипторы.
- **Итераторы и генераторы**: Для ленивой фильтрации и эффективной работы с коллекциями.
- **Асинхронность**: Использование `async`/`await`, `asyncio.Queue`, контекстных менеджеров.
- **Контрактное программирование**: Протоколы `TaskSource` и `TaskHandler` с `@runtime_checkable` обеспечивают гибкость и расширяемость.
- **Обработка исключений**: Создана иерархия собственных исключений для различных ошибок валидации.
- **Тестирование**: Используются `pytest` и `unittest` для покрытия функциональности (минимальное покрытие — 80%).
